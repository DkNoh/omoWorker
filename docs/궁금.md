## Q1. 스캐폴드 생성 코드를 건드리지 말라고 한 이유가 뭔가?

**질문 배경:** 보통은 스캐폴드로 기본 코드를 생성하고 개발자가 수동으로 수정하는데, 왜 건드리지 마라 했는지 이해가 안 됨.

**답변:**

두 가지 스캐폴드 철학이 있습니다.

**A. 생성 후 소유 (Rails/Django 방식 - 사용자 구상)**
- 스캐폴드로 기본 CRUD 생성 -> 개발자가 자유롭게 수정 -> 끝
- 재생성 안 함. 코드는 개발자 소유.
- 직관적, 유연, 대부분의 실무가 이 방식

**B. 재생성 안전 (Oracle 권장 - 제가 구현한 방식)**
- 생성 파일은 스캐폴드 소유, 개발자는 hook/Rules만 수정
- 재생성해도 커스텀 코드 보존
- 스캐폴드 개선 시 재생성으로 전파

Oracle이 B를 권고한 이유: 재생성이 위험하면 아무도 안 하고, 프레임워크가 개선을 전파하는 능력을 잃는다

하지만 폐쇄망 마이그레이션은 1회성이므로 재생성할 일이 거의 없음. 철학 A가 더 현실적.
철학 A를 택하면: Rules/hook/보존/게이트는 불필요. 스캐폴드의 가치는 초기 생성 품질에 집중.

## Q2. GlobalModelAdvice에서 4개 속성을 매 요청마다 채우는 게 표준인가?

**답변:**

네, Spring Boot + Thymeleaf SSR의 정석 패턴입니다.

- @ControllerAdvice + @ModelAttribute는 Spring 공식 메커니즘
- 모든 컨트롤러 요청에 대해 공통 Model 속성(user, menus, pageAuth, clientIp)을 주입
- Thymeleaf layout 프래그먼트(sidebar/header)가 이 속성을 사용해 HTML 렌더링
- Django(context processors), Rails(before_action) 등 다른 SSR 프레임워크도 동일한 패턴

## Q3. REST API 호출 시에도 GlobalModelAdvice가 실행되나?

**답변:**

네. @ControllerAdvice는 모든 컨트롤러의 모든 요청에 실행됩니다.

- /api/common-code/dept (JSON API) 호출시 GlobalModelAdvice가 실행되어 menus/pageAuth를 Model에 채움
- 하지만 @ResponseBody는 JSON을 직접 반환하므로 Model 속성을 전혀 사용하지 않음
- 세션 캐싱으로 DB 조회 비용은 제거됐지만, 개념적으로 불필요한 작업

개선 완료 (커밋 2a1f2fc): @ControllerAdvice(annotations = Controller.class)로 변경하여 @RestController에서는 GlobalModelAdvice가 실행되지 않도록 수정.

## Q4. API 요청 모델 스킵 - 공수와 효과

**답변:**

공수: 1줄 변경. @ControllerAdvice에 annotations = Controller.class 추가.

- @RestController (순수 JSON API)에서 GlobalModelAdvice 스킵
- @Controller (HTML 화면 + @ResponseBody 혼합)는 기존대로 실행
- 세션 캐싱으로 남은 @Controller 내부 @ResponseBody의 비용은 마이크로초 수준

원인 제거(스킵)와 증상 완화(캐싱) 둘 다 적용 완료.

## Q5. include refid=columns 가 빠지는 게 낫지 않나?

**답변:**

재사용 횟수에 따라 다릅니다.

- 2회 이상 재사용시 include가 맞음 (DRY, 드리프트 방지)
- 1회만 사용시 인라인이 나음 (간접 참조만 늘고 이득 없음)

SmsHistoryMapper의 columns는 selectList/selectListForExcel/selectDetail 3곳에서 사용하므로 include 유지가 정답.

스캐폴드는 baseQuery를 sql id=baseQuery와 include로 한 번 정의 후 3곳에서 참조. 동일 원칙.

## Q6. 폐쇄망 개발자는 AI 사용 불가, 나만 가능 → 철학 A가 맞나?

**질문 배경:** 레퍼런스를 완성해서 가져가도 폐쇄망 개발자들은 AI를 쓸 수 없음. 유일하게 나만 가능. 그럼 철학 A(생성 후 소유)로 가야 하는가?

**답변:**

맞습니다. 이 조건이면 **철학 A가 확정**입니다. 이유:

**핵심 제약: 개발자가 AI 없이 코드를 읽고 수정해야 함**

- 철학 B(hook/Rules)는 "AI가 생성 파일을 관리하고, 개발자는 hook만 채운다"는 전제
- 하지만 개발자가 AI 없이 직접 코드를 수정해야 하는 환경에서는:
  - hook 시스템을 이해하는 것 자체가 **학습 부담**
  - "왜 Service에서 rules.validateOnCreate를 호출하는데 정작 로직은 Rules 클래스에?" → 혼란
  - 직접 Service를 열고 수정하는 것이 **압도적으로 직관적**

**철학 A 방향성:**

스캐폴드의 목표가 바뀝니다:

| 철학 B (재생성 안전) | 철학 A (생성 후 소유) |
|---|---|
| 스캐폴드 = 관리 프레임워크 | 스캐폴드 = **고품질 출발점 생성기** |
| 개발자는 hook만 수정 | 개발자가 **생성 코드를 직접 수정** |
| 재생성으로 개선 전파 | 개선은 **다음 버전 스캐폴드로** (새 화면 생성 시) |
| Rules/hook/보존/게이트 필요 | **불필요 — 과잉 설계** |

**방향성 변화:**

1. **생성 품질에 올인** — 개발자가 손을 덜 대도록, 한 번에 더 완성된 코드를 생성
2. **TODO 주석으로 가이드** — 개발자가 수정해야 할 곳에 명확한 TODO 표시 ("여기에 업무 검증 추가")
3. **Rules/hook 제거 검토** — Service에 직접 validate 로직을 작성하는 패턴이 더 직관적
4. **보존 로직/드리프트 게이트는 낮은 우선순위** — 재생성을 안 할 거니 불필요
5. **문서가 핵심** — AI 없이 개발자가 이해할 수 있는 screen-generation-guide.md 절차서가 가장 중요한 자산

**즉, 스캐폴드는 "AI가 대신 만들어주는 고품질 템플릿"이 되고, 개발자는 그 위에 업무 로직을 얹는 구조.**
