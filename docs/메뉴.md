# 메뉴관리 화면 설계 메모

## 결론

사용자별 직접 권한이 아니라 **역할(Role) 기반 권한**으로 설계한다.

현재 운영 기준 테이블은 `EMP`, `DEP`만 존재하므로, v3 메뉴/권한 관리를 위해 다음 4개 테이블을 추가한다.

```text
TB_ROLE       역할
TB_EMP_ROLE   직원(EMP_ID + DEP_ID) ↔ 역할 매핑
TB_MENU       메뉴 트리 / URL
TB_MENU_AUTH  역할별 메뉴 기능 권한
```

`EMP.PERM_*`는 v3 권한 판단에 사용하지 않는다. 사용자는 항상 `(EMP_ID, DEP_ID)`로 식별한다.

## 권한 단위

권한은 화면 메뉴 단위로 나누고, URL suffix 규칙으로 API 접근까지 검증한다.

이 문서의 권한은 **화면/메뉴 접근 권한**이다. 승인 업무의 결재선, 승인 요청 상태, 승인 이력은 별도 승인 시스템에서 관리한다. 예를 들어 `CAN_APPROVE='Y'`는 `/approve` 버튼과 API를 호출할 수 있다는 뜻이지, 특정 건의 승인 순번/대리승인/반려 사유까지 보장한다는 뜻이 아니다. 그 업무 로직은 `docs/승인.md`의 승인 테이블과 Service 트랜잭션으로 검증한다.

| 권한 | 의미 |
|---|---|
| `CAN_READ` | 메뉴 노출, 화면 접근, 목록/상세 조회 |
| `CAN_CREATE` | 등록 |
| `CAN_UPDATE` | 수정 |
| `CAN_DELETE` | 삭제 |
| `CAN_APPROVE` | 승인/반려 |
| `CAN_CANCEL` | 취소성 업무 |
| `CAN_DOWNLOAD` | 엑셀/파일 다운로드 |
| `CAN_MASK_VIEW` | 개인정보 원문/비마스킹 조회 |

`CAN_READ`는 기본 전제 권한이다. 관리 화면에서는 `CREATE/UPDATE/DELETE/APPROVE/CANCEL/DOWNLOAD/MASK_VIEW` 중 하나를 켜면 `READ`도 자동으로 켜지게 한다.

## 관리 화면 구성

한 화면에 모든 기능을 넣지 말고 다음 화면 또는 탭으로 분리한다.

### 1. 메뉴 트리 관리

`TB_MENU`를 관리한다.

- 메뉴명
- 메뉴 ID
- 상위 메뉴
- URL
- 정렬순서
- 메뉴 타입: 그룹(`G`) / 화면 메뉴(`M`)
- 표시 여부(`DISPLAY_YN`)
- 사용 여부(`USE_YN`)
- 시스템 보호 여부(`SYSTEM_YN`)

규칙:

- 그룹 메뉴(`G`)는 URL이 없다.
- 화면 메뉴(`M`)는 URL이 필수다.
- 좌측 메뉴에는 `DISPLAY_YN='Y'`, `USE_YN='Y'`, `CAN_READ='Y'`인 화면 메뉴만 노출한다.
- 그룹 메뉴는 직접 권한을 부여하기보다, 읽을 수 있는 자식 메뉴가 있으면 자동 노출하는 방식이 좋다.
- `SYSTEM_YN='Y'`인 메뉴는 삭제하거나 URL을 바꾸지 못하게 막는다.

### 2. 역할 관리

`TB_ROLE`을 관리한다.

- 역할 코드: `ROLE_ADMIN`, `ROLE_MANAGER`, `ROLE_USER`, `ROLE_VIEWER` 등
- 역할명
- 설명
- 정렬순서
- 사용 여부

역할은 물리 삭제보다 `USE_YN='N'` 처리로 비활성화한다.

### 3. 역할별 메뉴 권한 매트릭스

`TB_MENU_AUTH`를 관리한다.

권장 UI:

```text
역할 선택: [ROLE_ADMIN v]

메뉴명              READ CREATE UPDATE DELETE APPROVE CANCEL DOWNLOAD MASK_VIEW
기본메뉴            [ ]  [ ]    [ ]    [ ]    [ ]     [ ]    [ ]      [ ]
  공지사항          [x]  [x]    [x]    [x]    [ ]     [ ]    [ ]      [ ]
  발송이력조회      [x]  [ ]    [ ]    [ ]    [ ]     [ ]    [x]      [x]
```

저장 시 `(MENU_ID, ROLE_CD)` 기준으로 `TB_MENU_AUTH`를 upsert한다.

권장 동작:

- `READ`를 끄면 나머지 권한도 함께 끈다.
- `READ`가 꺼진 메뉴는 좌측 메뉴에 노출하지 않는다.
- `DOWNLOAD`는 `/excel`, `/download`, `/export` 접근 권한이다.
- `MASK_VIEW`는 `/unmask` 또는 개인정보 원문 조회 권한이다.
- `APPROVE`, `CANCEL`은 승인/취소 업무가 있는 화면에서만 사용한다.
- `APPROVE`는 메뉴 권한일 뿐이다. 실제 승인 가능 여부는 승인 요청의 현재 단계, 승인자, 상태값을 승인 시스템 테이블에서 다시 판단한다.

### 4. 사용자 역할 부여

`TB_EMP_ROLE`을 관리한다.

흐름:

```text
EMP + DEP 조회 → 사용자 선택 → 역할 체크 → TB_EMP_ROLE 저장
```

권장 UI:

- 사번/이름/부서 검색
- 사용자 선택
- 역할 목록 체크박스
- 사용 여부

사용자에게 메뉴 권한을 직접 부여하지 않는다. 직접 권한을 열면 권한 추적이 어려워진다.

## URL suffix 권한 매핑

현재 v3 권한 검증은 메뉴 URL과 요청 URL suffix를 기준으로 판단한다.

| URL 패턴 | 필요 권한 |
|---|---|
| 화면 URL, `/data`, `/search`, `/detail`, `/tree` | `CAN_READ` |
| `/create`, `/register` | `CAN_CREATE` |
| `/update` | `CAN_UPDATE` |
| `/delete` | `CAN_DELETE` |
| `/approve`, `/reject` | `CAN_APPROVE` |
| `/cancel` | `CAN_CANCEL` |
| `/excel`, `/download`, `/export` | `CAN_DOWNLOAD` |
| `/unmask` | `CAN_MASK_VIEW` |

Controller에서 권한을 중복 검사하지 않는다. 실제 URL/API 차단은 `MenuAuthInterceptor`가 담당한다.

단, 승인 업무 Service는 별도 업무 검증을 수행한다. `MenuAuthInterceptor`는 “이 사용자가 승인 API에 접근 가능한 역할인가”만 판단하고, “이 사용자가 이 승인 건의 현재 승인자인가”는 승인 Service가 판단한다.

## 구현 순서

1. `TB_ROLE`, `TB_EMP_ROLE`, `TB_MENU`, `TB_MENU_AUTH` DDL/seed 확정
2. `ROLE_ADMIN`에게 전체 권한 seed
3. 메뉴 트리 관리 화면 생성
4. 역할 관리 화면 생성
5. 역할별 메뉴 권한 매트릭스 화면 생성
6. 사용자 역할 부여 화면 생성
7. `sms.menu.source=db`, `sms.role.source=db`로 실제 검증

## 테이블 설계 및 스크립트

전제:

- Schema: `SMS`
- Oracle 19c 기준
- 기존 사용자/부서 테이블: `SMS.EMP`, `SMS.DEP`
- 사용자 식별 키: `(EMP_ID, DEP_ID)`
- 운영 반영 DDL에는 `DROP TABLE`을 넣지 않는다.

### 2. `TB_EMP_ROLE`

직원과 역할의 매핑 테이블이다. 권한은 직원에게 직접 주지 않고 역할에 부여한다.

```sql
CREATE TABLE SMS.TB_EMP_ROLE (
    EMP_ID    VARCHAR2(12) NOT NULL,
    DEP_ID    VARCHAR2(12) NOT NULL,
    ROLE_CD   VARCHAR2(30) NOT NULL,
    USE_YN    CHAR(1)      DEFAULT 'Y' NOT NULL,
    REG_ID    VARCHAR2(12) NOT NULL,
    REG_DTTM  TIMESTAMP(6) DEFAULT SYSTIMESTAMP NOT NULL,
    UPD_ID    VARCHAR2(12),
    UPD_DTTM  TIMESTAMP(6),
    CONSTRAINT PK_TB_EMP_ROLE PRIMARY KEY (EMP_ID, DEP_ID, ROLE_CD),
    CONSTRAINT FK_TB_EMP_ROLE_EMP FOREIGN KEY (EMP_ID, DEP_ID)
        REFERENCES SMS.EMP (EMP_ID, DEP_ID),
    CONSTRAINT FK_TB_EMP_ROLE_ROLE FOREIGN KEY (ROLE_CD)
        REFERENCES SMS.TB_ROLE (ROLE_CD),
    CONSTRAINT CK_TB_EMP_ROLE_USE_YN CHECK (USE_YN IN ('Y', 'N'))
);

CREATE INDEX SMS.IX_TB_EMP_ROLE_01 ON SMS.TB_EMP_ROLE (ROLE_CD);
CREATE INDEX SMS.IX_TB_EMP_ROLE_02 ON SMS.TB_EMP_ROLE (DEP_ID, EMP_ID);

COMMENT ON TABLE SMS.TB_EMP_ROLE IS 'v3 직원-역할 매핑';
COMMENT ON COLUMN SMS.TB_EMP_ROLE.EMP_ID IS '직원 ID. EMP.EMP_ID';
COMMENT ON COLUMN SMS.TB_EMP_ROLE.DEP_ID IS '부서 ID. EMP.DEP_ID';
COMMENT ON COLUMN SMS.TB_EMP_ROLE.ROLE_CD IS '역할 코드. TB_ROLE.ROLE_CD';
```

관리자 역할 부여 예시:

```sql
MERGE INTO SMS.TB_EMP_ROLE T
USING (
    SELECT '관리자사번' EMP_ID, '관리자부서' DEP_ID, 'ROLE_ADMIN' ROLE_CD FROM DUAL
) S
ON (T.EMP_ID = S.EMP_ID AND T.DEP_ID = S.DEP_ID AND T.ROLE_CD = S.ROLE_CD)
WHEN MATCHED THEN UPDATE SET
    T.USE_YN = 'Y',
    T.UPD_ID = 'SYSTEM',
    T.UPD_DTTM = SYSTIMESTAMP
WHEN NOT MATCHED THEN INSERT (
    EMP_ID, DEP_ID, ROLE_CD, USE_YN, REG_ID
) VALUES (
    S.EMP_ID, S.DEP_ID, S.ROLE_CD, 'Y', 'SYSTEM'
);
```

### 4. `TB_MENU_AUTH`

역할별 메뉴 기능 권한 테이블이다.

```sql
CREATE TABLE SMS.TB_MENU_AUTH (
    MENU_ID        VARCHAR2(50) NOT NULL,
    ROLE_CD        VARCHAR2(30) NOT NULL,
    CAN_READ       CHAR(1)      DEFAULT 'N' NOT NULL,
    CAN_CREATE     CHAR(1)      DEFAULT 'N' NOT NULL,
    CAN_UPDATE     CHAR(1)      DEFAULT 'N' NOT NULL,
    CAN_DELETE     CHAR(1)      DEFAULT 'N' NOT NULL,
    CAN_APPROVE    CHAR(1)      DEFAULT 'N' NOT NULL,
    CAN_CANCEL     CHAR(1)      DEFAULT 'N' NOT NULL,
    CAN_DOWNLOAD   CHAR(1)      DEFAULT 'N' NOT NULL,
    CAN_MASK_VIEW  CHAR(1)      DEFAULT 'N' NOT NULL,
    USE_YN         CHAR(1)      DEFAULT 'Y' NOT NULL,
    REG_ID         VARCHAR2(12) NOT NULL,
    REG_DTTM       TIMESTAMP(6) DEFAULT SYSTIMESTAMP NOT NULL,
    UPD_ID         VARCHAR2(12),
    UPD_DTTM       TIMESTAMP(6),
    CONSTRAINT PK_TB_MENU_AUTH PRIMARY KEY (MENU_ID, ROLE_CD),
    CONSTRAINT FK_TB_MENU_AUTH_MENU FOREIGN KEY (MENU_ID)
        REFERENCES SMS.TB_MENU (MENU_ID),
    CONSTRAINT FK_TB_MENU_AUTH_ROLE FOREIGN KEY (ROLE_CD)
        REFERENCES SMS.TB_ROLE (ROLE_CD),
    CONSTRAINT CK_TB_MENU_AUTH_READ CHECK (CAN_READ IN ('Y', 'N')),
    CONSTRAINT CK_TB_MENU_AUTH_CREATE CHECK (CAN_CREATE IN ('Y', 'N')),
    CONSTRAINT CK_TB_MENU_AUTH_UPDATE CHECK (CAN_UPDATE IN ('Y', 'N')),
    CONSTRAINT CK_TB_MENU_AUTH_DELETE CHECK (CAN_DELETE IN ('Y', 'N')),
    CONSTRAINT CK_TB_MENU_AUTH_APPROVE CHECK (CAN_APPROVE IN ('Y', 'N')),
    CONSTRAINT CK_TB_MENU_AUTH_CANCEL CHECK (CAN_CANCEL IN ('Y', 'N')),
    CONSTRAINT CK_TB_MENU_AUTH_DOWNLOAD CHECK (CAN_DOWNLOAD IN ('Y', 'N')),
    CONSTRAINT CK_TB_MENU_AUTH_MASK CHECK (CAN_MASK_VIEW IN ('Y', 'N')),
    CONSTRAINT CK_TB_MENU_AUTH_USE_YN CHECK (USE_YN IN ('Y', 'N')),
    CONSTRAINT CK_TB_MENU_AUTH_READ_DEP CHECK (
        CAN_READ = 'Y'
        OR (
            CAN_CREATE = 'N'
            AND CAN_UPDATE = 'N'
            AND CAN_DELETE = 'N'
            AND CAN_APPROVE = 'N'
            AND CAN_CANCEL = 'N'
            AND CAN_DOWNLOAD = 'N'
            AND CAN_MASK_VIEW = 'N'
        )
    )
);

CREATE INDEX SMS.IX_TB_MENU_AUTH_01 ON SMS.TB_MENU_AUTH (ROLE_CD, USE_YN);

COMMENT ON TABLE SMS.TB_MENU_AUTH IS 'v3 역할별 메뉴 권한';
COMMENT ON COLUMN SMS.TB_MENU_AUTH.CAN_READ IS '화면 접근 및 조회 권한';
COMMENT ON COLUMN SMS.TB_MENU_AUTH.CAN_CREATE IS '신규 등록 권한';
COMMENT ON COLUMN SMS.TB_MENU_AUTH.CAN_UPDATE IS '수정 권한';
COMMENT ON COLUMN SMS.TB_MENU_AUTH.CAN_DELETE IS '삭제 권한';
COMMENT ON COLUMN SMS.TB_MENU_AUTH.CAN_APPROVE IS '승인/반려 API 접근 권한';
COMMENT ON COLUMN SMS.TB_MENU_AUTH.CAN_CANCEL IS '취소 권한';
COMMENT ON COLUMN SMS.TB_MENU_AUTH.CAN_DOWNLOAD IS '다운로드 권한';
COMMENT ON COLUMN SMS.TB_MENU_AUTH.CAN_MASK_VIEW IS '개인정보 비마스킹 조회 권한';
```

`ROLE_ADMIN` 전체 권한 seed:

```sql
INSERT INTO SMS.TB_MENU_AUTH (
    MENU_ID, ROLE_CD,
    CAN_READ, CAN_CREATE, CAN_UPDATE, CAN_DELETE,
    CAN_APPROVE, CAN_CANCEL, CAN_DOWNLOAD, CAN_MASK_VIEW,
    USE_YN, REG_ID

SELECT
    MENU_ID,
    'ROLE_ADMIN',
    'Y', 'Y', 'Y', 'Y',
    'Y', 'Y', 'Y', 'Y',
    'Y', 'SYSTEM'
FROM SMS.TB_MENU
WHERE USE_YN = 'Y';
```

```sql
        'N' CAN_APPROVE,
        'N' CAN_CANCEL,
        'Y' CAN_DOWNLOAD,
        'N' CAN_MASK_VIEW
    FROM DUAL
) S
ON (T.MENU_ID = S.MENU_ID AND T.ROLE_CD = S.ROLE_CD)
WHEN MATCHED THEN UPDATE SET
    T.CAN_READ = S.CAN_READ,
    T.CAN_CREATE = S.CAN_CREATE,
    T.CAN_UPDATE = S.CAN_UPDATE,
    T.CAN_DELETE = S.CAN_DELETE,
    T.CAN_APPROVE = S.CAN_APPROVE,
    T.CAN_CANCEL = S.CAN_CANCEL,
    T.CAN_DOWNLOAD = S.CAN_DOWNLOAD,
    T.CAN_MASK_VIEW = S.CAN_MASK_VIEW,
    T.USE_YN = 'Y',
    T.UPD_ID = 'SYSTEM',
    T.UPD_DTTM = SYSTIMESTAMP
WHEN NOT MATCHED THEN INSERT (
    MENU_ID, ROLE_CD,
    CAN_READ, CAN_CREATE, CAN_UPDATE, CAN_DELETE,
    CAN_APPROVE, CAN_CANCEL, CAN_DOWNLOAD, CAN_MASK_VIEW,
    USE_YN, REG_ID
) VALUES (
    S.MENU_ID, S.ROLE_CD,
    S.CAN_READ, S.CAN_CREATE, S.CAN_UPDATE, S.CAN_DELETE,
    S.CAN_APPROVE, S.CAN_CANCEL, S.CAN_DOWNLOAD, S.CAN_MASK_VIEW,
    'Y', 'SYSTEM'
);
```

### 5. 실행 순서

```text
1. TB_ROLE 생성
2. TB_MENU 생성
3. TB_EMP_ROLE 생성
4. TB_MENU_AUTH 생성
5. TB_ROLE 기본 역할 seed
6. TB_MENU 메뉴 seed
7. TB_EMP_ROLE 관리자 역할 부여
8. TB_MENU_AUTH ROLE_ADMIN 전체 권한 부여
9. sms.menu.source=db, sms.role.source=db 전환 후 검증
```

## 확장 판단

지금은 `TB_BUTTON_AUTH`, `TB_API_AUTH` 같은 세부 테이블을 만들지 않는다.

현재 URL suffix 정책이면 대부분의 화면 권한은 충분히 커버된다. 나중에 같은 화면 안에서 특수 버튼이 많고 버튼마다 독립 권한이 필요해지면 그때 `TB_MENU_ACTION` 같은 확장 테이블을 추가한다.

즉, 현재 단계의 목표는 다음이다.

```text
스캐폴드가 생성한 화면 URL을 TB_MENU에 등록하고,
역할별로 READ/CREATE/UPDATE/DELETE/APPROVE/CANCEL/DOWNLOAD/MASK_VIEW를 관리할 수 있게 한다.
```

## 권한관리 화면 구성안

현재 화면에는 메뉴/역할/권한을 관리하는 전용 화면이 없다. 추가한다면 좌측 메뉴의 가장 아래 `시스템관리` 그룹에 다음 메뉴를 붙인다.

```text
시스템관리
  메뉴관리        /system/menu-manage
  역할관리        /system/role-manage
  권한관리        /system/auth-manage
  사용자역할관리  /system/emp-role-manage
```

초기 구현은 화면을 너무 쪼개기보다 `권한관리` 한 화면에서 탭으로 시작하는 편이 좋다. 운영자가 실제로 자주 쓰는 흐름은 “역할 선택 → 메뉴별 권한 체크 → 저장”이기 때문이다.

### 1. 권한관리 메인 화면

URL: `/system/auth-manage`

구성:

```text
[역할 선택]
ROLE_ADMIN   시스템 관리자
ROLE_MANAGER 업무 관리자
ROLE_USER    일반 사용자
ROLE_VIEWER  조회 사용자

[권한 매트릭스]
메뉴명             READ CREATE UPDATE DELETE APPROVE CANCEL DOWNLOAD MASK_VIEW
기본메뉴           [ ]  [ ]    [ ]    [ ]    [ ]     [ ]    [ ]      [ ]
  공지사항         [x]  [x]    [x]    [x]    [ ]     [ ]    [ ]      [ ]
SMS발송조회        [ ]  [ ]    [ ]    [ ]    [ ]     [ ]    [ ]      [ ]
  발송이력조회     [x]  [ ]    [ ]    [ ]    [ ]     [ ]    [x]      [x]

[저장]
```

저장 대상은 `TB_MENU_AUTH`다. 저장 기준은 `(MENU_ID, ROLE_CD)`이며, 기존 행이 있으면 update, 없으면 insert한다.

필수 동작:

- 역할을 선택하면 해당 역할의 메뉴 권한을 조회한다.
- `READ`를 끄면 나머지 권한도 자동으로 끈다.
- `CREATE/UPDATE/DELETE/APPROVE/CANCEL/DOWNLOAD/MASK_VIEW` 중 하나를 켜면 `READ`도 자동으로 켠다.
- 그룹 메뉴(`MENU_TYPE='G'`)는 직접 저장하지 않아도 된다. 자식 중 읽을 수 있는 메뉴가 있으면 좌측 메뉴에 자동 노출한다.
- `SYSTEM_YN='Y'` 메뉴는 삭제/URL 변경 대상이 아니지만 권한 부여 대상은 될 수 있다.

### 2. 메뉴관리 탭 또는 화면

URL: `/system/menu-manage`

대상 테이블: `TB_MENU`

구성:

```text
[메뉴 트리]
기본메뉴
SMS발송조회
캠페인SMS
시스템관리

[상세]
메뉴 ID
상위 메뉴
메뉴명
URL
메뉴 타입(G/M)
정렬순서
표시 여부
사용 여부
시스템 보호 여부
```

필수 동작:

- 그룹 메뉴는 URL을 입력하지 않는다.
- 화면 메뉴는 URL을 필수로 입력한다.
- `MENU_URL`은 중복을 허용하지 않는다.
- `SYSTEM_YN='Y'`는 URL과 메뉴 ID를 수정하지 못하게 한다.
- 신규 업무 화면을 스캐폴드로 만든 뒤 이 화면에서 `TB_MENU`에 등록한다.

### 3. 역할관리 탭 또는 화면

URL: `/system/role-manage`

대상 테이블: `TB_ROLE`

구성:

```text
역할 코드     역할명        사용 여부  정렬
ROLE_ADMIN   시스템 관리자  Y         10
ROLE_MANAGER 업무 관리자    Y         20
ROLE_USER    일반 사용자    Y         30
ROLE_VIEWER  조회 사용자    Y         40
```

필수 동작:

- 역할 코드는 `ROLE_` prefix를 사용한다.
- 역할은 물리 삭제하지 않고 `USE_YN='N'`으로 비활성화한다.
- 사용 중인 역할을 비활성화할 때는 해당 역할을 가진 사용자 수를 보여준다.

### 4. 사용자역할관리 탭 또는 화면

URL: `/system/emp-role-manage`

대상 테이블: `TB_EMP_ROLE`

구성:

```text
[사용자 검색]
사번 / 이름 / 부서

[사용자 선택 결과]
EMP_ID  EMP_NM  DEP_ID  DEP_NM

[역할 체크]
[x] ROLE_ADMIN
[ ] ROLE_MANAGER
[x] ROLE_USER
```

필수 동작:

- 사용자는 `EMP + DEP` 기준으로 검색한다.
- 저장 기준은 `(EMP_ID, DEP_ID, ROLE_CD)`다.
- 사용자에게 메뉴 권한을 직접 부여하지 않는다.
- dev/prod에서 로그인하려면 사용자의 활성 역할이 최소 1개 있어야 한다.

### 5. 초기 구현 우선순위

처음부터 4개 화면을 모두 만들 필요는 없다. 최소 구현은 다음 순서가 좋다.

```text
1. 권한관리 /system/auth-manage
   - 역할 선택
   - 메뉴 권한 매트릭스
   - TB_MENU_AUTH 저장

2. 사용자역할관리 /system/emp-role-manage
   - EMP 조회
   - 역할 부여
   - TB_EMP_ROLE 저장

3. 메뉴관리 /system/menu-manage
   - 신규 화면 URL 등록
   - 메뉴 노출/사용 여부 관리

4. 역할관리 /system/role-manage
   - 역할 추가/비활성화
```

현재 단계에서 가장 먼저 필요한 화면은 `권한관리`와 `사용자역할관리`다. 테이블과 seed를 넣은 뒤 이 두 화면이 있어야 운영자가 DB 메뉴 전환 후 권한을 직접 조정할 수 있다.
