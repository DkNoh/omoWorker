# 승인 시스템 설계 메모

## 결론

승인 시스템은 메뉴 권한과 분리해서 설계한다.

메뉴 권한의 `CAN_APPROVE`는 승인 화면과 `/approve`, `/reject` API에 접근할 수 있는 **기능 권한**이다. 실제 승인 가능 여부는 승인 요청의 상태, 현재 단계, 승인자, 대리 승인 여부, 반려/취소 상태를 승인 테이블에서 다시 판단해야 한다.

```text
메뉴 권한: 이 사용자가 승인 버튼/API를 사용할 수 있는가?
승인 시스템: 이 사용자가 이 건의 현재 승인자인가?
```

따라서 승인 업무는 다음 두 조건을 모두 만족해야 한다.

1. 역할에 `CAN_APPROVE='Y'`가 있다.
2. 승인 요청 테이블상 현재 사용자가 해당 건의 승인 대상자다.

## 설계 범위

현재 운영 기준으로 `EMP`, `DEP`만 존재한다고 가정한다. 승인 시스템은 v3 신규 테이블로 추가한다.

승인 대상 업무는 캠페인 발송, 발송대상 변경, 메시지 변경 등으로 확장될 수 있으므로 특정 업무 테이블에 종속되지 않는 공통 승인 모델을 둔다.

## 추천 테이블

### 1. `TB_APPROVAL_REQUEST`

승인 요청의 헤더 테이블이다. 승인 대상 업무 1건당 1행을 만든다.

| 컬럼 | 설명 |
|---|---|
| `APPROVAL_ID` | 승인 요청 ID. PK |
| `BIZ_TYPE` | 승인 업무 유형. 예: `CAMPAIGN_SMS`, `TARGET_UPLOAD`, `MESSAGE_TEMPLATE` |
| `BIZ_KEY` | 승인 대상 업무 키. 예: 캠페인 ID, 템플릿 ID |
| `TITLE` | 승인 제목 |
| `REQUEST_STATUS` | 요청 상태 |
| `REQUEST_EMP_ID` | 요청자 EMP_ID |
| `REQUEST_DEP_ID` | 요청자 DEP_ID |
| `REQUEST_DTTM` | 요청일시 |
| `CURRENT_STEP` | 현재 승인 단계 |
| `FINAL_DECISION_DTTM` | 최종 승인/반려/취소 일시 |
| `REMARK` | 요청 메모 |
| `REG_ID`, `REG_DTTM`, `UPD_ID`, `UPD_DTTM` | 등록/수정 감사 컬럼 |

권장 상태값:

| 상태 | 의미 |
|---|---|
| `DRAFT` | 임시 저장. 아직 승인 요청 전 |
| `REQUESTED` | 승인 요청됨 |
| `IN_PROGRESS` | 다단계 승인 진행 중 |
| `APPROVED` | 최종 승인 |
| `REJECTED` | 반려 |
| `CANCELED` | 요청 취소 |

`BIZ_TYPE + BIZ_KEY`는 같은 업무 건의 중복 승인 요청을 막기 위해 상태 조건부 유니크 정책을 검토한다. Oracle에서는 함수 기반 인덱스 또는 Service 검증으로 처리한다.

### 2. `TB_APPROVAL_STEP`

승인 단계와 승인자를 저장한다. 결재선 1명당 1행이다.

| 컬럼 | 설명 |
|---|---|
| `APPROVAL_ID` | 승인 요청 ID. FK |
| `STEP_NO` | 승인 단계 번호 |
| `APPROVER_EMP_ID` | 승인자 EMP_ID |
| `APPROVER_DEP_ID` | 승인자 DEP_ID |
| `APPROVER_ROLE_CD` | 역할 기준 승인일 때 선택 사용 |
| `STEP_STATUS` | 단계 상태 |
| `DECISION_CD` | 승인/반려 결정 코드 |
| `DECISION_COMMENT` | 승인/반려 의견 |
| `DECISION_DTTM` | 승인/반려 일시 |
| `DELEGATE_EMP_ID` | 대리 승인자 EMP_ID. 선택 |
| `DELEGATE_DEP_ID` | 대리 승인자 DEP_ID. 선택 |
| `REG_ID`, `REG_DTTM`, `UPD_ID`, `UPD_DTTM` | 등록/수정 감사 컬럼 |

기본키는 `(APPROVAL_ID, STEP_NO, APPROVER_EMP_ID, APPROVER_DEP_ID)`를 권장한다.

권장 단계 상태값:

| 상태 | 의미 |
|---|---|
| `WAIT` | 아직 차례가 아님 |
| `PENDING` | 현재 승인 대기 |
| `APPROVED` | 해당 단계 승인 완료 |
| `REJECTED` | 해당 단계 반려 |
| `SKIPPED` | 조건상 생략 |
| `CANCELED` | 요청 취소로 종료 |

### 3. `TB_APPROVAL_HISTORY`

승인 상태 변경 이력 테이블이다. 요청, 승인, 반려, 취소, 재요청 등 모든 이벤트를 남긴다.

| 컬럼 | 설명 |
|---|---|
| `HIST_ID` | 이력 ID. PK |
| `APPROVAL_ID` | 승인 요청 ID. FK |
| `STEP_NO` | 관련 단계 번호. 요청/취소는 null 가능 |
| `ACTION_CD` | `REQUEST`, `APPROVE`, `REJECT`, `CANCEL`, `REOPEN` 등 |
| `BEFORE_STATUS` | 변경 전 상태 |
| `AFTER_STATUS` | 변경 후 상태 |
| `ACTOR_EMP_ID` | 행위자 EMP_ID |
| `ACTOR_DEP_ID` | 행위자 DEP_ID |
| `ACTION_COMMENT` | 사유/의견 |
| `ACTION_DTTM` | 행위 일시 |
| `CLIENT_IP` | 행위 IP |

승인 시스템은 현재 상태만으로 판단하지 않고, 감사/추적을 위해 이력을 반드시 남긴다.

### 4. 선택 테이블: `TB_APPROVAL_RULE`

업무 유형별 기본 결재선을 DB로 관리하고 싶으면 추가한다. 초기에는 과할 수 있으므로 2단계로 둔다.

| 컬럼 | 설명 |
|---|---|
| `BIZ_TYPE` | 업무 유형 |
| `STEP_NO` | 단계 번호 |
| `APPROVER_TYPE` | `EMP`, `ROLE`, `DEP_MANAGER` 등 |
| `APPROVER_VALUE` | 승인자 ID 또는 역할 코드 |
| `USE_YN` | 사용 여부 |

초기 구현은 화면/Service에서 승인자를 직접 선택하게 하고, 승인 패턴이 안정되면 `TB_APPROVAL_RULE`로 자동 결재선을 도입하는 것을 권장한다.

## 테이블 설계 및 스크립트

전제:

- Schema: `SMS`
- Oracle 19c 기준
- 기존 사용자/부서 테이블: `SMS.EMP`, `SMS.DEP`
- 승인 테이블은 `EMP_ID + DEP_ID` 복합키로 사용자/부서를 참조한다.
- 운영 반영 DDL에는 `DROP TABLE`을 넣지 않는다.

### 1. `TB_APPROVAL_REQUEST`

승인 요청 헤더 테이블이다.

```sql
CREATE TABLE SMS.TB_APPROVAL_REQUEST (
    APPROVAL_ID          VARCHAR2(40)  NOT NULL,
    BIZ_TYPE             VARCHAR2(50)  NOT NULL,
    BIZ_KEY              VARCHAR2(100) NOT NULL,
    TITLE                VARCHAR2(200) NOT NULL,
    REQUEST_STATUS       VARCHAR2(20)  DEFAULT 'DRAFT' NOT NULL,
    REQUEST_EMP_ID       VARCHAR2(12)  NOT NULL,
    REQUEST_DEP_ID       VARCHAR2(12)  NOT NULL,
    REQUEST_DTTM         TIMESTAMP(6),
    CURRENT_STEP         NUMBER(5)     DEFAULT 0 NOT NULL,
    FINAL_DECISION_DTTM  TIMESTAMP(6),
    REMARK               VARCHAR2(1000),
    REG_ID               VARCHAR2(12)  NOT NULL,
    REG_DTTM             TIMESTAMP(6)  DEFAULT SYSTIMESTAMP NOT NULL,
    UPD_ID               VARCHAR2(12),
    UPD_DTTM             TIMESTAMP(6),
    CONSTRAINT PK_TB_APPROVAL_REQUEST PRIMARY KEY (APPROVAL_ID),
    CONSTRAINT FK_TB_APPROVAL_REQ_EMP FOREIGN KEY (REQUEST_EMP_ID, REQUEST_DEP_ID)
        REFERENCES SMS.EMP (EMP_ID, DEP_ID),
    CONSTRAINT CK_TB_APPROVAL_REQ_STATUS CHECK (
        REQUEST_STATUS IN ('DRAFT', 'REQUESTED', 'IN_PROGRESS', 'APPROVED', 'REJECTED', 'CANCELED')
    )
);

CREATE INDEX SMS.IX_TB_APPROVAL_REQ_01
    ON SMS.TB_APPROVAL_REQUEST (BIZ_TYPE, BIZ_KEY);

CREATE INDEX SMS.IX_TB_APPROVAL_REQ_02
    ON SMS.TB_APPROVAL_REQUEST (REQUEST_EMP_ID, REQUEST_DEP_ID, REQUEST_DTTM DESC);

CREATE INDEX SMS.IX_TB_APPROVAL_REQ_03
    ON SMS.TB_APPROVAL_REQUEST (REQUEST_STATUS, CURRENT_STEP);

COMMENT ON TABLE SMS.TB_APPROVAL_REQUEST IS 'v3 승인 요청 헤더';
COMMENT ON COLUMN SMS.TB_APPROVAL_REQUEST.APPROVAL_ID IS '승인 요청 ID';
COMMENT ON COLUMN SMS.TB_APPROVAL_REQUEST.BIZ_TYPE IS '승인 업무 유형';
COMMENT ON COLUMN SMS.TB_APPROVAL_REQUEST.BIZ_KEY IS '승인 대상 업무 키';
COMMENT ON COLUMN SMS.TB_APPROVAL_REQUEST.REQUEST_STATUS IS '승인 요청 상태';
COMMENT ON COLUMN SMS.TB_APPROVAL_REQUEST.CURRENT_STEP IS '현재 승인 단계';
```

중복 진행 방지 인덱스 예시:

```sql
CREATE UNIQUE INDEX SMS.UX_TB_APPROVAL_REQ_ACTIVE
ON SMS.TB_APPROVAL_REQUEST (
    BIZ_TYPE,
    BIZ_KEY,
    CASE
        WHEN REQUEST_STATUS IN ('DRAFT', 'REQUESTED', 'IN_PROGRESS') THEN 'ACTIVE'
        ELSE APPROVAL_ID
    END
);
```

의미: 같은 `BIZ_TYPE + BIZ_KEY`에 대해 진행 중인 승인 요청은 1건만 허용한다. 승인 완료/반려/취소 건은 이력으로 남고 새 요청을 만들 수 있다.

### 2. `TB_APPROVAL_STEP`

승인 단계와 승인자를 저장한다.

```sql
CREATE TABLE SMS.TB_APPROVAL_STEP (
    APPROVAL_ID       VARCHAR2(40) NOT NULL,
    STEP_NO           NUMBER(5)    NOT NULL,
    APPROVER_EMP_ID   VARCHAR2(12) NOT NULL,
    APPROVER_DEP_ID   VARCHAR2(12) NOT NULL,
    APPROVER_ROLE_CD  VARCHAR2(30),
    STEP_STATUS       VARCHAR2(20) DEFAULT 'WAIT' NOT NULL,
    DECISION_CD       VARCHAR2(20),
    DECISION_COMMENT  VARCHAR2(1000),
    DECISION_DTTM     TIMESTAMP(6),
    DELEGATE_EMP_ID   VARCHAR2(12),
    DELEGATE_DEP_ID   VARCHAR2(12),
    REG_ID            VARCHAR2(12) NOT NULL,
    REG_DTTM          TIMESTAMP(6) DEFAULT SYSTIMESTAMP NOT NULL,
    UPD_ID            VARCHAR2(12),
    UPD_DTTM          TIMESTAMP(6),
    CONSTRAINT PK_TB_APPROVAL_STEP PRIMARY KEY (
        APPROVAL_ID, STEP_NO, APPROVER_EMP_ID, APPROVER_DEP_ID
    ),
    CONSTRAINT FK_TB_APPROVAL_STEP_REQ FOREIGN KEY (APPROVAL_ID)
        REFERENCES SMS.TB_APPROVAL_REQUEST (APPROVAL_ID),
    CONSTRAINT FK_TB_APPROVAL_STEP_EMP FOREIGN KEY (APPROVER_EMP_ID, APPROVER_DEP_ID)
        REFERENCES SMS.EMP (EMP_ID, DEP_ID),
    CONSTRAINT FK_TB_APPROVAL_STEP_DELEGATE FOREIGN KEY (DELEGATE_EMP_ID, DELEGATE_DEP_ID)
        REFERENCES SMS.EMP (EMP_ID, DEP_ID),
    CONSTRAINT FK_TB_APPROVAL_STEP_ROLE FOREIGN KEY (APPROVER_ROLE_CD)
        REFERENCES SMS.TB_ROLE (ROLE_CD),
    CONSTRAINT CK_TB_APPROVAL_STEP_STATUS CHECK (
        STEP_STATUS IN ('WAIT', 'PENDING', 'APPROVED', 'REJECTED', 'SKIPPED', 'CANCELED')
    ),
    CONSTRAINT CK_TB_APPROVAL_STEP_DECISION CHECK (
        DECISION_CD IS NULL OR DECISION_CD IN ('APPROVE', 'REJECT', 'SKIP', 'CANCEL')
    )
);

CREATE INDEX SMS.IX_TB_APPROVAL_STEP_01
    ON SMS.TB_APPROVAL_STEP (APPROVER_EMP_ID, APPROVER_DEP_ID, STEP_STATUS);

CREATE INDEX SMS.IX_TB_APPROVAL_STEP_02
    ON SMS.TB_APPROVAL_STEP (APPROVAL_ID, STEP_NO, STEP_STATUS);

COMMENT ON TABLE SMS.TB_APPROVAL_STEP IS 'v3 승인 단계/승인자';
COMMENT ON COLUMN SMS.TB_APPROVAL_STEP.STEP_NO IS '승인 단계 번호';
COMMENT ON COLUMN SMS.TB_APPROVAL_STEP.APPROVER_EMP_ID IS '승인자 EMP_ID';
COMMENT ON COLUMN SMS.TB_APPROVAL_STEP.APPROVER_DEP_ID IS '승인자 DEP_ID';
COMMENT ON COLUMN SMS.TB_APPROVAL_STEP.STEP_STATUS IS '승인 단계 상태';
```

승인함 조회 핵심 조건:

```sql
SELECT
    R.APPROVAL_ID,
    R.BIZ_TYPE,
    R.BIZ_KEY,
    R.TITLE,
    R.REQUEST_EMP_ID,
    R.REQUEST_DEP_ID,
    R.REQUEST_DTTM,
    S.STEP_NO,
    S.STEP_STATUS
FROM SMS.TB_APPROVAL_REQUEST R
JOIN SMS.TB_APPROVAL_STEP S
  ON S.APPROVAL_ID = R.APPROVAL_ID
WHERE S.STEP_STATUS = 'PENDING'
  AND (
      (S.APPROVER_EMP_ID = #{empId} AND S.APPROVER_DEP_ID = #{depId})
      OR (S.DELEGATE_EMP_ID = #{empId} AND S.DELEGATE_DEP_ID = #{depId})
  )
ORDER BY R.REQUEST_DTTM DESC;
```

### 3. `TB_APPROVAL_HISTORY`

승인 이벤트 이력 테이블이다.

```sql
CREATE TABLE SMS.TB_APPROVAL_HISTORY (
    HIST_ID          VARCHAR2(40)  NOT NULL,
    APPROVAL_ID      VARCHAR2(40)  NOT NULL,
    STEP_NO          NUMBER(5),
    ACTION_CD        VARCHAR2(20)  NOT NULL,
    BEFORE_STATUS    VARCHAR2(20),
    AFTER_STATUS     VARCHAR2(20)  NOT NULL,
    ACTOR_EMP_ID     VARCHAR2(12)  NOT NULL,
    ACTOR_DEP_ID     VARCHAR2(12)  NOT NULL,
    ACTION_COMMENT   VARCHAR2(1000),
    ACTION_DTTM      TIMESTAMP(6)  DEFAULT SYSTIMESTAMP NOT NULL,
    CLIENT_IP        VARCHAR2(45),
    CONSTRAINT PK_TB_APPROVAL_HISTORY PRIMARY KEY (HIST_ID),
    CONSTRAINT FK_TB_APPROVAL_HIST_REQ FOREIGN KEY (APPROVAL_ID)
        REFERENCES SMS.TB_APPROVAL_REQUEST (APPROVAL_ID),
    CONSTRAINT FK_TB_APPROVAL_HIST_EMP FOREIGN KEY (ACTOR_EMP_ID, ACTOR_DEP_ID)
        REFERENCES SMS.EMP (EMP_ID, DEP_ID),
    CONSTRAINT CK_TB_APPROVAL_HIST_ACTION CHECK (
        ACTION_CD IN ('DRAFT', 'REQUEST', 'APPROVE', 'REJECT', 'CANCEL', 'REOPEN', 'SKIP')
    )
);

CREATE INDEX SMS.IX_TB_APPROVAL_HIST_01
    ON SMS.TB_APPROVAL_HISTORY (APPROVAL_ID, ACTION_DTTM);

CREATE INDEX SMS.IX_TB_APPROVAL_HIST_02
    ON SMS.TB_APPROVAL_HISTORY (ACTOR_EMP_ID, ACTOR_DEP_ID, ACTION_DTTM DESC);

COMMENT ON TABLE SMS.TB_APPROVAL_HISTORY IS 'v3 승인 상태 변경 이력';
COMMENT ON COLUMN SMS.TB_APPROVAL_HISTORY.HIST_ID IS '승인 이력 ID';
COMMENT ON COLUMN SMS.TB_APPROVAL_HISTORY.ACTION_CD IS '승인 이벤트 코드';
COMMENT ON COLUMN SMS.TB_APPROVAL_HISTORY.BEFORE_STATUS IS '변경 전 상태';
COMMENT ON COLUMN SMS.TB_APPROVAL_HISTORY.AFTER_STATUS IS '변경 후 상태';
```

이력 저장 예시:

```sql
INSERT INTO SMS.TB_APPROVAL_HISTORY (

    ACTOR_EMP_ID, ACTOR_DEP_ID,
    ACTION_COMMENT, CLIENT_IP
) VALUES (
    #{histId}, #{approvalId}, #{stepNo}, #{actionCd},
    #{beforeStatus}, #{afterStatus},
    #{actorEmpId}, #{actorDepId},
    #{actionComment}, #{clientIp}

```

### 4. 선택 테이블 `TB_APPROVAL_RULE`

업무 유형별 기본 결재선 규칙 테이블이다. 초기에는 선택 사항이다.

```sql
CREATE TABLE SMS.TB_APPROVAL_RULE (
    BIZ_TYPE        VARCHAR2(50) NOT NULL,
    STEP_NO         NUMBER(5)    NOT NULL,
    APPROVER_TYPE   VARCHAR2(20) NOT NULL,
    APPROVER_VALUE  VARCHAR2(50) NOT NULL,
    SORT_ORD        NUMBER(5)    DEFAULT 0 NOT NULL,
    USE_YN          CHAR(1)      DEFAULT 'Y' NOT NULL,
    REG_ID          VARCHAR2(12) NOT NULL,
    REG_DTTM        TIMESTAMP(6) DEFAULT SYSTIMESTAMP NOT NULL,
    UPD_ID          VARCHAR2(12),
    UPD_DTTM        TIMESTAMP(6),
    CONSTRAINT PK_TB_APPROVAL_RULE PRIMARY KEY (BIZ_TYPE, STEP_NO, APPROVER_TYPE, APPROVER_VALUE),
    CONSTRAINT CK_TB_APPROVAL_RULE_TYPE CHECK (
        APPROVER_TYPE IN ('EMP', 'ROLE', 'DEP_MANAGER')
    ),
    CONSTRAINT CK_TB_APPROVAL_RULE_USE_YN CHECK (USE_YN IN ('Y', 'N'))
);

CREATE INDEX SMS.IX_TB_APPROVAL_RULE_01
    ON SMS.TB_APPROVAL_RULE (BIZ_TYPE, USE_YN, SORT_ORD);

COMMENT ON TABLE SMS.TB_APPROVAL_RULE IS 'v3 업무 유형별 기본 결재선 규칙';
COMMENT ON COLUMN SMS.TB_APPROVAL_RULE.BIZ_TYPE IS '승인 업무 유형';
COMMENT ON COLUMN SMS.TB_APPROVAL_RULE.APPROVER_TYPE IS '승인자 산정 방식: EMP, ROLE, DEP_MANAGER';
COMMENT ON COLUMN SMS.TB_APPROVAL_RULE.APPROVER_VALUE IS '승인자 값: EMP_ID 또는 ROLE_CD 등';
```

규칙 seed 예시:

```sql
INSERT INTO SMS.TB_APPROVAL_RULE (
    BIZ_TYPE, STEP_NO, APPROVER_TYPE, APPROVER_VALUE, SORT_ORD, USE_YN, REG_ID
) VALUES (
    'CAMPAIGN_SMS', 1, 'ROLE', 'ROLE_MANAGER', 10, 'Y', 'SYSTEM'
);

INSERT INTO SMS.TB_APPROVAL_RULE (
    BIZ_TYPE, STEP_NO, APPROVER_TYPE, APPROVER_VALUE, SORT_ORD, USE_YN, REG_ID
) VALUES (
    'CAMPAIGN_SMS', 2, 'ROLE', 'ROLE_ADMIN', 20, 'Y', 'SYSTEM'
);
```

### 5. 업무 테이블 연결 컬럼 예시

승인 대상 업무 테이블에는 승인 요청과 상태를 연결할 컬럼이 필요하다.

예: 캠페인 테이블이 `SMS.CAMPAIGN`이라고 확정된 경우

```sql
ALTER TABLE SMS.CAMPAIGN ADD (
    APPROVAL_ID      VARCHAR2(40),
    APPROVAL_STATUS  VARCHAR2(20) DEFAULT 'DRAFT' NOT NULL
);

ALTER TABLE SMS.CAMPAIGN ADD CONSTRAINT FK_CAMPAIGN_APPROVAL
    FOREIGN KEY (APPROVAL_ID)
    REFERENCES SMS.TB_APPROVAL_REQUEST (APPROVAL_ID);

ALTER TABLE SMS.CAMPAIGN ADD CONSTRAINT CK_CAMPAIGN_APPROVAL_STATUS CHECK (
    APPROVAL_STATUS IN ('DRAFT', 'REQUESTED', 'IN_PROGRESS', 'APPROVED', 'REJECTED', 'CANCELED')
);

CREATE INDEX SMS.IX_CAMPAIGN_APPROVAL_01
    ON SMS.CAMPAIGN (APPROVAL_STATUS, APPROVAL_ID);
```

실제 업무 테이블명과 PK는 폐쇄망 DB 확인 후 확정한다.

### 6. 실행 순서

```text
1. TB_APPROVAL_REQUEST 생성
2. TB_APPROVAL_STEP 생성
3. TB_APPROVAL_HISTORY 생성
4. TB_APPROVAL_RULE 생성 여부 결정 후 필요 시 생성
5. 승인 대상 업무 테이블에 APPROVAL_ID / APPROVAL_STATUS 추가
6. 업무 유형별 승인 rule seed 작성
7. 승인 요청/승인/반려/취소 Service 구현
```

## 승인 흐름

### 1. 승인 요청

```text
업무 데이터 임시 저장
→ 승인자/결재선 선택
→ TB_APPROVAL_REQUEST 생성
→ TB_APPROVAL_STEP 생성
→ 첫 단계 STEP_STATUS = PENDING
→ 이력 REQUEST 저장
```

업무 테이블에는 승인 상태 컬럼을 둔다.

예:

```text
APPROVAL_STATUS = DRAFT / REQUESTED / APPROVED / REJECTED / CANCELED
APPROVAL_ID     = TB_APPROVAL_REQUEST.APPROVAL_ID
```

업무 테이블과 승인 테이블 변경은 같은 Service 트랜잭션으로 묶는다.

### 2. 승인

```text
CAN_APPROVE 권한 확인
→ 현재 PENDING 단계 조회
→ 현재 사용자가 승인자인지 확인
→ STEP_STATUS = APPROVED
→ 다음 단계가 있으면 다음 STEP_STATUS = PENDING
→ 다음 단계가 없으면 REQUEST_STATUS = APPROVED, 업무 상태도 APPROVED
→ 이력 APPROVE 저장
```

### 3. 반려

```text
CAN_APPROVE 권한 확인
→ 현재 PENDING 단계 조회
→ 현재 사용자가 승인자인지 확인
→ STEP_STATUS = REJECTED
→ REQUEST_STATUS = REJECTED
→ 업무 상태도 REJECTED
→ 이력 REJECT 저장
```

반려 시 사유 입력은 필수로 둔다.

### 4. 요청 취소

```text
CAN_CANCEL 또는 요청자 본인 여부 확인
→ 아직 최종 승인 전인지 확인
→ REQUEST_STATUS = CANCELED
→ 남은 STEP_STATUS = CANCELED
→ 업무 상태도 CANCELED
→ 이력 CANCEL 저장
```

취소 권한은 업무 정책에 따라 요청자 본인만 허용할지, `CAN_CANCEL` 권한자도 허용할지 정한다.

## 권한과 승인 로직의 경계

| 구분 | 담당 |
|---|---|
| 승인 화면 접근 | `TB_MENU_AUTH.CAN_READ` |
| 승인 버튼 노출 | `TB_MENU_AUTH.CAN_APPROVE` |
| `/approve`, `/reject` API 접근 | `MenuAuthInterceptor` + `CAN_APPROVE` |
| 해당 건 승인자 여부 | 승인 Service + `TB_APPROVAL_STEP` |
| 승인 상태 전이 | 승인 Service 트랜잭션 |
| 이력 저장 | `TB_APPROVAL_HISTORY` |

즉, `CAN_APPROVE`만 있다고 모든 승인 건을 승인할 수 있으면 안 된다. 반드시 현재 단계의 승인자 또는 허용된 대리 승인자인지 확인한다.

## 화면 구성

### 승인 요청 화면

- 업무 내용 요약
- 승인자 선택 또는 결재선 선택
- 요청 의견
- 요청 버튼

### 승인함 화면

- 내가 승인해야 할 건
- 요청자, 요청부서, 업무유형, 제목, 요청일시
- 승인/반려 버튼
- 반려 사유 입력

### 승인 이력 화면

- 업무 건별 승인 진행 상태
- 단계별 승인자/처리일시/의견
- 전체 이력 타임라인

### 관리자 화면

- 업무 유형별 기본 결재선 관리. `TB_APPROVAL_RULE` 도입 후 생성

## Service 설계 원칙

- 승인/반려/취소는 반드시 Service 계층 `@Transactional`로 처리한다.
- Controller는 요청/응답과 화면 반환에 집중한다.
- 승인 가능 여부는 Service에서 확인한다.
- update 결과가 0건이면 충돌로 보고 실패 처리한다.
- 상태 전이는 enum 또는 상수로 한 곳에서 관리한다.
- 승인 이력 저장 실패를 무시하지 않는다.

## 최소 구현 순서

1. `TB_APPROVAL_REQUEST`, `TB_APPROVAL_STEP`, `TB_APPROVAL_HISTORY` DDL 작성
2. 업무 테이블에 `APPROVAL_ID`, `APPROVAL_STATUS` 연결 방식 확정
3. 승인 요청 Service 작성
4. 승인함 조회 Service 작성
5. 승인/반려 Service 작성
6. 승인 이력 조회 화면 작성
7. 메뉴 권한의 `CAN_APPROVE`, `CAN_CANCEL`과 실제 승인 Service 검증을 연결

## 아직 결정할 것

- 승인자는 직접 지정할지, 역할 기준으로 지정할지
- 1단계 승인만 필요한지, 다단계 승인이 필요한지
- 요청자가 승인 취소를 할 수 있는 시점
- 반려 후 재요청을 새 `APPROVAL_ID`로 만들지, 기존 요청을 `REOPEN`할지
- 대리 승인 허용 여부
- 업무별 승인 상태 컬럼명이 이미 존재하는지
